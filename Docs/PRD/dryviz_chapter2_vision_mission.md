# Chapter 2: Vision & Mission

## 2.1 Vision Statement

Dry Viz aims to become a leading platform for learning programming through interactive visualization. Instead of asking students to memorize syntax and algorithms, Dry Viz helps them understand how programs actually execute by turning code into a visual, step-by-step experience.

> To make programming understandable through intelligent visualization.

---

## 2.2 Mission Statement

Our mission is to simplify programming education by helping students understand code through visualization instead of memorization — transforming source code into step-by-step animations, AI-powered explanations, execution traces, and complexity analysis.

> Help every student learn programming by seeing how code actually works.

---

## 2.3 Why Dry Viz Exists

Programming is difficult for beginners largely because most educational platforms explain concepts through static text and code samples. Students often recognize what code looks like without understanding what happens when it runs. Dry Viz exists to close that gap: instead of reading code, students watch it execute and follow every step.

---

## 2.4 Core Beliefs

These are the pedagogical assumptions Dry Viz is built on — the *why* behind the product.

1. **Visualization beats text.** People build intuition faster from watching execution than from reading about it. Animation, memory visualization, and algorithm visualization should be defaults, not add-ons.
2. **Understanding over memorization.** Students should know *why* an algorithm works, not just recall its code — real understanding is what transfers to problem-solving.
3. **Students learn differently.** Some read, some watch, some experiment. Dry Viz combines all three modes rather than picking one.
4. **AI should teach, not replace thinking.** AI acts as a mentor — it should guide students toward the reasoning behind a solution rather than handing over the answer immediately.

---

## 2.5 Product Principles

These are the operating rules for building features — the *how*, distinct from the beliefs above.

1. **Visualize before explaining** — show what happens, then explain why.
2. **Every visualization is interactive by default** — replay, pause, and step-through are built in, not optional extras.
3. **Show complexity, don't just state it** — time and space complexity are demonstrated visually, not only given as notation.
4. **Support exploration** — students should be able to experiment with code without fear of "breaking" anything.
5. **Simplicity over feature bloat** — a clear experience beats a crowded one.
6. **Accuracy over impressiveness** — visualizations and explanations must correctly represent actual execution, even when a shortcut would look flashier.

## 2.7 Success Vision

When a student thinks "I don't understand this algorithm," Dry Viz should be the first thing they open — the way developers default to GitHub for hosting or LeetCode for practice.

---

## 2.8 Closing Statement

Dry Viz isn't built to replace coding platforms — it's built to improve how students learn from them. The goal is to become the visual learning layer that helps programmers actually understand code execution, algorithms, and computational thinking.
